01

Cluster Architecture

The CASL cluster consists of a single bare-metal compute server named pasteur and a login virtual machine named pasteur-login that runs on top of it. You always connect to the login node — never directly to the compute node.

KVM Virtual Machine

Access All users

// key concept
You submit jobs from pasteur-login. Slurm schedules them to run on pasteur. Your files live on pasteur's drives and are accessible from the login node via NFS mounts.
02

Storage Layout

All persistent storage lives on pasteur's 14TB /data drive and is accessible from pasteur-login via NFS. Each storage area has a specific purpose — use the right one to stay within quota and keep things organized.

/data/ (14TB — on pasteur)
├── home/ → /mnt/datahome on pasteur-login
│ └── <username>/ 5 GB quota shell config, SSH keys, scripts
├── apps/ → /apps on pasteur-login
│ └── casl/
│ ├── common/ maintainer write 100 GB (group) shared conda envs + tools, read-only for all
│ └── <username>/ 15 GB quota your conda envs, personal software
├── scratch/ → /scratch on pasteur-login
│ └── casl/
│ ├── common/ maintainer write shared datasets, read-only for all
│ └── <username>/ 50 GB quota job input/output, temporary data
└── shared/ → /shared on pasteur-login
└── projects/ project-based access (by request)

What Goes Where

Item Location Why
.bashrc, SSH keys, scripts ~/ (home) Small config, always needed
Conda environments /apps/casl/<user>/ Persistent, not job data
Job input/output data /scratch/casl/<user>/ Temporary, job-specific
Shared conda envs /apps/casl/common/ Maintained by arachchige/igaribay
Project data (auth'd) /shared/projects/<name>/ Request access from admin
// warning
Do not store conda environments or large datasets in your home directory. The 5GB home quota will fill immediately and your login shell may break.
03

Logging In

Connect via SSH to the login node. You will land directly in your home directory on the NFS-backed storage.

// SSH from your workstation
ssh <username>@10.196.20.50
// What you see after login
# Your prompt — home is on NFS-backed storage
<username>@pasteur-login:~$ pwd
/mnt/datahome/<username>

# Your storage areas
<username>@pasteur-login:~$ ls /apps/casl/<username>/    # your apps
<username>@pasteur-login:~$ ls /scratch/casl/<username>/  # your scratch
<username>@pasteur-login:~$ ls /shared/                   # shared resources

Setting Up Conda

1
Check shared environments first
Admin-maintained envs live in /apps/casl/common/ — check if one already covers your needs
2
Create personal env in your apps directory
conda create --prefix /apps/casl/<user>/envs/myenv python=3.11
3
Activate and use
conda activate /apps/casl/<user>/envs/myenv
04

Running Jobs with Slurm

All compute work must be submitted as Slurm jobs. Do not run heavy computation directly on pasteur-login — it is a shared login node with limited resources.

Partitions

Partition Max Walltime Default Use Case Access
debug 1 hour no Testing, debugging, short runs Everyone
standard 48 hours yes ★ Regular training runs Everyone
extended 7 days no Long multi-GPU runs multigpu QOS required

QOS Levels

QOS (Quality of Service) controls your job's priority and GPU access. Default is normal. Contact admin to request elevated access.

Job Submission Examples

// minimal job — single GPU, standard partition
sbatch --partition=standard \
       --gres=gpu:h200:1 \
       --cpus-per-task=8 \
       --mem=32G \
       --time=12:00:00 \
       --job-name=my_train \
       my_script.sh
// debug job — quick test, no GPU
sbatch --partition=debug \
       --cpus-per-task=4 \
       --mem=8G \
       --time=00:30:00 \
       my_test.sh
// example job script — my_script.sh
#!/bin/bash
#SBATCH --partition=standard
#SBATCH --gres=gpu:h200:1
#SBATCH --cpus-per-task=8
#SBATCH --mem=32G
#SBATCH --time=12:00:00
#SBATCH --output=/scratch/casl/%u/logs/%j.out
#SBATCH --error=/scratch/casl/%u/logs/%j.err

# Activate your conda env
conda activate /apps/casl/<user>/envs/myenv

# Run your code
python train.py

Monitoring Jobs

// common slurm commands
squeue                          # all jobs in queue
squeue -u $USER                 # your jobs only
sinfo                           # partition and node status
scontrol show job <jobid>       # detailed job info
scancel <jobid>                 # cancel a job
sacct -u $USER                 # your job history
// tip — job states
PD = pending (waiting for resources)  |  R = running  |  CG = completing  |  CD = completed  |  F = failed

// quota increases
All quotas can be increased per request. Contact arachchige and explain your use case. Scratch and apps quotas are most commonly raised for large conda environments or dataset-heavy workflows.
06

Rules and Policies

Rule Detail
No jobs on login node Submit all compute via Slurm. pasteur-login is for job submission, file management, and light editing only.
1 GPU by default Request multi-GPU access from admin. Standard jobs are limited to 1 GPU.
Use scratch for job I/O Write job outputs to /scratch/casl/<user>/, not home or apps directories.
Shared spaces are read-only common/ dirs in scratch and apps are maintained by arachchige and igaribay only.
Project access by request Some datasets in /shared/projects/ require authorization. Request access from admin.
Clean up scratch regularly Scratch is not backed up. Move important results to home or a project folder promptly.
